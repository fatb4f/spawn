# Cross-Repo Analysis (Memory V0)

## Objective
Map where `agent:memory` touches the current fabric and identify which repos are authoritative vs integration-only.

## Repo Roles and Interfaces

| Repo | Role for Memory V0 | Provides | Consumes | Notes |
|---|---|---|---|---|
| `spawn` | Primary implementation repo | `spawnd`, `spawnctl`, proto API, contracts, generation scripts | codex session artifacts, systemd user services | Must become SSOT runtime owner for memory |
| `codex-dome` | Schema source candidate | Existing SSOT schema corpus (`codex-dome/ssot/schemas/*`), examples | N/A (for this scope) | Provides migration seed for minimal schema pack |
| `dotfiles` | Runtime wiring and deployment | systemd user units, presets, wrapper commands, spawn config | `spawn` CLI/daemon entrypoints | Must align with refactored `spawn` module paths and unit model |
| `fabric-docs` | Cross-repo standards | MUST rules, systemd guidelines, model docs | Inputs from `spawn`/`mesh` status | Contains stale watcher-era references that affect clarity |
| `mesh` | Out-of-scope runtime peer | ops-router scaffold | `fabric-docs` standards | Not required for Memory V0 delivery |
| `hydra-gen` | Methodology reference | spec-first architecture and ADR patterns | N/A | Useful for governance style, not a runtime dependency |

## Current `spawn` Reality vs Memory Target

## In place
- Modular source layout (`core`, `runtime`, `contracts`, `adapters`, CLI groups).
- Proto control-plane with generated stubs.
- Derived OpenAPI catalog from contract models.
- Codex session event ingestion + transient execution pattern.

## Implemented for memory
- Schema pack exists in `src/spawn/ssot/schemas/`.
- Schema registry + validator CLI exist (`spawnctl ssot *`).
- Memory contracts and persistence/rebuild pipeline are implemented.

## Cross-Repo Couplings (High Impact)

1. `spawn` <-> `dotfiles` systemd coupling
- `dotfiles/units/user/spawnd-codex-session-refresh.service` calls `spawnd codex-session-refresh`.
- Default config is generated by `spawnctl daemon write-config`; no static tracked `spawnd.toml` required.
- `dotfiles/units/user/spawn-api.service` no longer depends on a missing `spawn.socket`.

2. `spawn` <-> `codex-dome` schema lineage
- `codex-dome/ssot/schemas/` still remains the source lineage reference.
- Migrated minimal subset now lives under `src/spawn/ssot/schemas`.

3. `spawn` <-> `fabric-docs` policy coupling
- `spawn` claims compliance with MUST/systemd guidelines.
- `fabric-docs` still includes watcher-era components in `docs/COMPONENTS.md`, reducing architectural precision.

## Authoritative Ownership (Proposed)
- `spawn` owns:
  - runtime behavior,
  - memory SSOT schemas and validators,
  - proto control-plane.
- `codex-dome` contributes:
  - initial schema templates/examples only.
- `dotfiles` owns:
  - activation/preset wiring only (no business logic).
- `fabric-docs` owns:
  - cross-repo normative standards and migration guidance.
